# 依赖安装工作流

## 目标

以一致、低摩擦的方式处理缺失或需要新增的依赖。

## 适用场景

当出现以下情况时使用此技能：

- 构建、测试或运行时报错，提示缺少某个包
- 新增导入或功能需要安装依赖
- 需要同步 `package.json`、锁文件或工作区清单
- 代理需要决定是否安装、更新或验证依赖

## 工作流程

1. 识别项目结构与包管理器。

   - 优先读取根目录及子包目录中的锁文件与清单文件。
   - 如果存在 `pnpm-lock.yaml`，优先使用 `pnpm`。
   - 否则如果存在 `yarn.lock`，优先使用 `yarn`。
   - 否则如果存在 `package-lock.json`，优先使用 `npm`。
   - 如果存在多包目录结构，识别 `pnpm-workspace.yaml`、`workspaces` 字段或类似 workspace 配置。
   - 如果都不存在，检查 `package.json` 脚本、仓库约定和当前子目录位置。

2. 确认依赖确实需要安装。

   - 如果失败是由拼写错误、过时导入、错误相对路径或本地文件问题引起的，先修复这些问题。
   - 如果缺失的是 Node 内置模块，不要安装第三方包。
   - 如果包明确缺失，再进行安装。

3. 选择正确的安装范围。

   - 单包项目：直接在项目根目录安装。
   - Monorepo：优先在需要该依赖的 workspace 包内安装，除非该依赖应由根包统一管理。
   - 如果依赖只用于某个子包，不要错误安装到根目录。
   - 如果仓库已有统一依赖策略，遵循该策略。

4. 使用项目包管理器进行安装。

   - 使用检测到的包管理器对应的标准添加/安装命令来添加运行时依赖。
   - 使用对应的开发依赖参数添加仅开发工具。
   - 如果仓库是 monorepo，优先使用支持工作区的命令。
   - 如果需要跨 workspace 共享，优先明确目标 workspace。

5. 处理 peer dependency 问题。

   - 先判断 peer dependency 是缺失、版本不兼容，还是由现有锁文件导致。
   - 如果只是缺少 peer dependency，按项目策略补齐。
   - 如果是版本冲突，优先选择与现有主依赖兼容的版本，而不是盲目升级。
   - 如果修复会引入较大范围依赖变更，先说明风险再继续。

6. 验证结果。

   - 确认依赖已添加到正确的清单文件中。
   - 确认锁文件已更新。
   - 在合适的时候重新运行失败的命令。
   - 如果是 workspace 安装，确认目标 workspace 可正确解析该依赖。

7. 如果安装失败，按错误类型分流处理。

   - 版本冲突：检查当前主依赖与 peer 要求，选择兼容版本。
   - 仓库或网络问题：确认 registry、网络和镜像配置。
   - 权限问题：避免继续重试，先修复权限或环境。
   - workspace 解析问题：检查依赖是否装在错误层级。
   - 安装脚本失败：先读脚本输出，再决定是否重试。

8. 重试策略。

   - 只有在明确知道会改变结果时才重试。
   - 同一问题最多重试一次。
   - 重试前必须改变一个确定因素，例如安装范围、版本、包管理器参数或网络配置。
   - 如果仍然失败，停止并清晰说明阻塞原因。

## 决策规则

- 如果依赖缺失且修复方式明显，就自动安装。
- 如果包名有歧义，安装前先询问。
- 如果更改属于重大升级，安装前先询问。
- 如果命令会影响全局环境，安装前先询问。
- 如果安装需要移除或替换大量包，先询问再继续。
- 如果是 peer dependency 修复，优先保持最小变更。

## 命令选择

使用检测到的包管理器：

- `pnpm add <package>` / `pnpm add -D <package>`
- `yarn add <package>` / `yarn add -D <package>`
- `npm install <package>` / `npm install -D <package>`

Monorepo 常用形式：

- `pnpm --filter <workspace> add <package>`
- `yarn workspace <workspace> add <package>`
- `npm install --workspace=<workspace> <package>`

## 验证清单

- [ ] 已识别项目结构与包管理器
- [ ] 已确认安装范围正确
- [ ] 已选择正确的依赖类型
- [ ] 清单文件已更新
- [ ] 锁文件已更新
- [ ] 必要时已重新运行失败命令
- [ ] 没有改动无关文件
- [ ] peer dependency 问题已按最小变更原则处理

## 输出风格

汇报结果时请使用简短总结：

- 添加了什么依赖
- 为什么需要它
- 安装位置是根包还是 workspace 包
- 验证是否成功
- 是否还需要人工后续处理
